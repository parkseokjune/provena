# Phase 0 결과 — 가설 검증 통과 ✅

**질문:** 출력 코드 스팬을 의미 임베딩만으로 올바른 소스에 귀속할 수 있는가? 미근거 구간은 정직하게 걸러지는가?

**셋업:** 소스 3개(OAuth 스펙 / Stripe 웹훅 / 대화 요구사항) + Claude가 생성했다고 가정한 `auth.ts`(함수 6개, 그중 1개는 의도적으로 소스 없는 "모델 지식"). 로컬 임베딩 `all-MiniLM-L6-v2`, API 키 불필요.

## 두 지표

| 지표 | 결과 | 의미 |
|------|------|------|
| **A) 검색 (top-1 소스)** | **5/5 = 100%** | 근거 있는 모든 스팬이 올바른 소스를 1순위로 찾음 |
| **B) 결정 (고정 임계값)** | 5/6 = 83% | 경계값 1건 오분류 (검색은 맞았으나 임계값 미달) |

## 핵심 발견

1. **검색은 풀렸다 (100%).** 제품의 가장 어려운 부분 = "어느 소스에서 왔나"를 임베딩만으로 정확히 맞춤.
2. **유일한 미스는 임계값 경계 문제** — `buildAccessToken`은 올바른 `oauth-spec.md`를 1순위로 찾았으나(0.343) 고정 컷오프 0.35를 아슬아슬하게 못 넘어 UNGROUNDED로 오분류. 검색 실패가 아님.
3. **미근거 분리가 깨끗하다** — 모델 지식인 `slugify`는 0.079로 근거 스팬(0.38~0.64)과 확연히 분리됨. 환각 위험 플래그가 잘 작동.

## 결론 → Phase 2 설계가 옳음

고정 글로벌 임계값은 깨지기 쉬움. **Phase 2의 LLM 심판**이 이를 대체한다: 1순위 후보에 대해 "이 스팬이 이 소스에서 유래했나? 근거 문장을 인용하라"를 검증 → 경계값 문제 해소, 거짓양성 억제.
